# 向量数据库改造方案（归档 / 待实施）

> 状态：**待实施**
> 触发条件：当视频条目数量突破单工作区 **1 万条** 量级，或需要接入 AI 模糊搜索时启动本方案。
> 当前阶段：仍使用 `.psp/media_cache.json` 的扁平 KV 结构，足以应对小规模检索。

---

## 一、改造目标

把 `MediaCache` 从 **纯结构化字典** 升级为 **结构化元数据 + 向量索引** 的双层存储，为以下能力做铺垫：

- 自然语言模糊搜索（"找出去年去日本旅游那段视频"）
- 基于语义的相似视频推荐
- 后续接入字幕 / AI 摘要后的全文语义检索

---

## 二、整体架构

```
┌─────────────────────────────────────────┐
│ 元数据层（真源, source of truth）       │
│ .psp/media_cache.json                   │
│   Key   = 相对路径                      │
│   Value = { Alias, Tags, ..., Hash }    │
└──────────────┬──────────────────────────┘
               │ 派生 / 可重建
               ▼
┌─────────────────────────────────────────┐
│ 向量索引层（派生数据）                  │
│ .psp/vectors.bin       连续 float32 数组 │
│ .psp/vectors.idx.json  行号 ↔ Key 映射   │
└─────────────────────────────────────────┘
```

**核心原则：向量库永远是派生数据，丢了能从元数据重建，避免数据一致性地狱。**

---

## 三、本地向量库选型

按落地难度从低到高：

| 方案 | 形态 | 适配性 | 适用规模 |
|------|------|--------|----------|
| **自己写 + 暴力余弦** | `vectors.bin` + 内存数组 | ★★★★★ | < 几万条 |
| **HNSWlib.Net / FAISS.NET** | NuGet 包 | ★★★★ | 几十万条 |
| **SQLite + sqlite-vss / sqlite-vec** | 单文件 db | ★★★ | 想要 SQL + 向量混合检索 |
| **LiteDB + 自建索引** | 纯托管 | ★★★ | 不想引入原生依赖 |
| **Qdrant / Milvus 本地服务** | 独立进程 | ★★ | 不推荐，杀鸡用牛刀 |

**首选方案 1**：JSON 存元数据，`.psp/vectors.bin` 存浮点数组，启动时一次性读入内存做暴力 cosine。

---

## 四、Embedding 来源

| 来源 | 优点 | 缺点 |
|------|------|------|
| **OpenAI / 通义 / 豆包 embedding API** | 效果好、零本地负担 | 要联网、要 Key、要钱 |
| **本地 ONNX 模型**（bge-small-zh、m3e-small 等） | 离线、免费 | 首次加载几百MB，需要 `Microsoft.ML.OnnxRuntime` |
| **本地 llama.cpp / Ollama embedding** | 离线、可换模型 | 要拉外部进程 |

中文模糊搜索建议 `bge-small-zh-v1.5`（512维）或 `bge-m3`（1024维）。

**实施策略：先用云 API 跑通流程，再切本地模型。**

## 五、需要改造的模块

### 5.1 数据结构层（`MediaCache.cs`）

`MediaCacheEntry` 增加字段：

| 字段 | 用途 |
|------|------|
| `EmbeddingVersion` | 当前条目的 embedding 模型/文本版本号 |
| `EmbeddingHash` | 参与 embedding 的源文本 SHA1，用于判断是否要重新算 |

**注意**：向量本身**不要**塞进 JSON（几百维浮点放 JSON 里又大又慢）。

### 5.2 新增向量存储模块

文件位置：`.psp/vectors.bin` + `.psp/vectors.idx.json`

| 文件 | 内容 |
|------|------|
| `vectors.bin` | 连续 `float32[]`，按行存 |
| `vectors.idx.json` | 行号 → 相对路径Key 映射 + 模型名 + 维度 |

接口设计：

```csharp
public interface IVectorStore
{
    void Upsert(string key, float[] vector);
    void Remove(string key);
    IEnumerable<(string Key, float Score)> Search(float[] queryVec, int topK, Func<string, bool> filter = null);
    void Save();
    void Load();
}
```

### 5.3 新增 Embedding 服务抽象

```csharp
public interface IEmbeddingProvider
{
    string ModelName { get; }
    int Dimension { get; }
    Task<float[]> EmbedAsync(string text);
    Task<float[][]> EmbedBatchAsync(string[] texts);
}
```

实现优先级：
1. `OpenAIEmbeddingProvider`（先跑通）
2. `OnnxLocalEmbeddingProvider`（离线方案）

配置项写入用户文档目录的 `config.ini`：

```ini
EmbeddingProvider=openai
EmbeddingModel=text-embedding-3-small
EmbeddingApiKey=...        ; 建议 DPAPI 加密后写
EmbeddingDim=1536
AllowCloudEmbedding=false  ; 默认关闭，需用户主动开启
```

### 5.4 索引构建 / 同步流程

**触发点：**

| 时机 | 动作 |
|------|------|
| 工作区首次打开 | 扫描差异，对没有向量或 `EmbeddingHash` 不匹配的条目批量 embedding |
| 用户改别名/标签 | 标记该条目 dirty，下次空闲或保存时增量 embedding |
| 文件被删除 | 同步从向量库移除 |

**实现要点：**

- 后台线程 + 任务队列，不卡 UI
- 批量调用 embedding（OpenAI 一次最多 2048 条）
- 失败重试 + 断点续跑（记录最后一条成功的 Key）

### 5.5 检索层

```csharp
Task<List<SearchResult>> SearchAsync(string query, int topK = 20, Filter filter = null);
```

执行流程：

1. 对 query 做 embedding
2. 暴力 cosine 算 topK
3. 可叠加结构化过滤（如"只在标签包含『教程』的子集里搜"）

**强烈建议混合检索**：向量相似度 + BM25/关键字 加权融合，纯向量对短查询不稳定。

### 5.6 UI 层

- 搜索框输入 → debounce 300ms → 调 `SearchAsync` → 列表展示，命中字段高亮
- 结果项右侧显示相似度分值（调试用，可隐藏）

## 六、改造路线图（分 4 步走，每步可独立验证）

| 步骤 | 目标 | 验收标准 |
|------|------|----------|
| **1. 抽象元数据层** | `MediaCacheEntry` 增加 hash/version 字段；提供"取可检索文本"的方法 | 不影响现有功能 |
| **2. 接入 Embedding API** | 实现 `IEmbeddingProvider`，命令行能把一个工作区全量 embedding 出来，落 `vectors.bin` | 文件正确生成、能加载 |
| **3. 实现搜索** | 暴力 cosine 检索 + UI 搜索框 | 能用自然语言搜出相关视频 |
| **4. 增量同步 + 本地模型** | dirty 队列 + ONNX 本地 embedding 切换 | 离线可用，改标签后能即时被搜到 |

## 七、容易踩的坑

1. **维度/模型变更要全量重建**
   在 `vectors.idx.json` 里记录模型名和维度，启动时校验，不一致直接删旧的重建。

2. **路径作为 Key 的稳定性**
   用户重命名文件夹会导致 Key 失效。可以同时存一份"文件内容 hash"（前 64KB 即可）作为兜底身份。

3. **不要每次启动全量 embedding**
   用 `EmbeddingHash`（源文本 SHA1）做 diff，仅对变化的条目重算。

4. **大文件不要 embedding 文件内容**
   视频本身别 embedding，embedding 元信息 + 用户字段就够；以后接字幕再扩展。

5. **隐私**
   走云 API 时要让用户知情。`config.ini` 中 `AllowCloudEmbedding=false` 默认关闭，需用户主动开启。

6. **批处理与限流**
   云 API 有 RPS 限制，需要做令牌桶 / 队列限速；本地模型则要限制并发 batch size 防爆显存/内存。
